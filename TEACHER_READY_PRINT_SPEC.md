# Teacher-Ready Print Spec

## North Star

Make every printed product look like a document a teacher could hand to a student or use in class tomorrow: clean hierarchy, real spacing, no UI chrome, no JSON, just a real document.

## Definition

A teacher-ready document:

- prints cleanly on standard paper
- reads like a classroom handout, assessment, or planning document
- uses document hierarchy instead of application layout
- contains only teacher-useful metadata
- avoids internal system labels and AI-flavored artifacts

## Scope

This spec applies to the print route and print rendering surface currently implemented in:

- `src/components_new/v4/ProductViewer.tsx`
- `src/components_new/v4/PrintProductPage.tsx`
- `src/components_new/v4/v4.css`

## Hard Requirements

### 1. Document-first layout

Printed products must not look like app cards.

Acceptance criteria:

- no borders, shadows, rounded cards, or dashboard panels
- no action buttons, toolbars, or controls in print output
- no JSON keys, debug labels, or internal identifiers
- content reads top-to-bottom like a document, not side-by-side like an app shell
- no overflow clipping, scroll containers, or truncated content in print preview

### 2. Product-specific pagination

Each product type must respect its natural reading and teaching flow.

Acceptance criteria:

- tests: page break before each major section or concept cluster; keep item stem and response area together
- lessons: page break before major phases such as warm-up, concept introduction, guided practice, independent practice, and exit ticket
- reviews: page break before rationale, review points, and practice prompts where present
- instructional maps: page break before concepts, units, anchors, and relationships where present
- no single assessment item or multi-step activity should split awkwardly across pages when avoidable

### 3. Typography and spacing

Printed products must use stable document typography.

Acceptance criteria:

- title: 20px to 24px, bold, centered
- section headers: 14px to 16px, bold, left-aligned
- body text: 12px to 13px with line height between 1.3 and 1.4
- margins: 1 inch on all sides for print
- spacing: 16px to 24px between sections; 8px to 12px between items
- use document typography, not app display fonts, on the print surface

### 4. Minimal useful metadata

The header should help a teacher orient themselves quickly.

Acceptance criteria:

- include document title
- include grade and subject when available in payload or session context
- include generated date
- optionally include “Generated from teacher documents”
- do not include productId, unitId, documentId, sessionId, or raw system fields

### 5. Expected writing space

Documents must leave room where teachers and students expect to write.

Acceptance criteria:

- tests: 3 to 5 blank lines under short-answer items; more space for constructed response
- lessons: space for teacher notes, activity work, and exit tickets where applicable
- reviews: space under practice prompts
- response-space styling must remain printable and not collapse under page breaks
- long-form prompts should receive more writing space than short recall prompts

### 6. Teacher-only content labeling

Teacher guidance may appear, but must be clearly framed.

Acceptance criteria:

- labels such as “Teacher Notes”, “Answer Guidance”, and “Scaffolds” are visually distinct
- teacher-only content is readable in print and does not overpower student-facing content
- teacher-only sections should never appear as raw metadata dumps

### 7. No AI or internal-product artifacts

Printed documents should feel authored, not computed.

Acceptance criteria:

- no “Generated by AI” copy
- no confidence scores or pipeline diagnostics
- no raw cognitive tags unless the product explicitly requires teacher mode metadata
- no internal IDs or developer vocabulary

## Product Notes

### Tests

- prioritize item readability over density
- keep prompts and answer space together
- optional answer guidance belongs at the end or in clearly labeled teacher-only sections

### Lessons

- sequence should feel teachable from top to bottom
- instructional phases should be visually obvious
- teacher notes should sit near the relevant activity, not in a separate debug panel

### Reviews

- review points should scan quickly
- practice prompts need enough whitespace for written work

### Instructional maps

- preserve concept and relationship clarity over visual novelty
- tables and relationship blocks should avoid splitting when printed

## QA Checklist

- open a saved product via `/print/:productId`
- verify Chrome print preview, Safari print preview, and browser PDF export before signoff
- confirm no app chrome appears in print preview
- confirm title and section hierarchy are obvious on the first page
- confirm no internal IDs or JSON labels are visible
- confirm page breaks happen at product-appropriate boundaries
- confirm stems and response areas stay together where possible
- confirm teacher-only content is clearly labeled
- confirm the document would be usable in class without manual cleanup
